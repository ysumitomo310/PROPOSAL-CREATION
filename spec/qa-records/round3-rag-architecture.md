# Round 3: RAGアーキテクチャ選定 - Q&A記録

> **ステータス**: 完了
> **日付**: 2026-02-17

## Q1. グラフDB + ベクトルDBの構成

- **決定**: Neo4j統合型（グラフDB + ベクトルインデックス内蔵）
- **理由**: 300-700ノード規模ではグラフ探索がファーストクラスの操作であるNeo4jが最適。ベクトル検索性能もこの規模では十分。Cypher + Vector検索のワンクエリ実行が可能。

## Q2. Embeddingモデルの選定

- **決定**: OpenAI text-embedding-3-large（3,072次元）、差し替え可能設計
- **理由**: 日本語+英語+ERP略語の混在テキストに対する多言語理解力。この規模ではEmbeddingモデルの差よりAgentic RAGのロジック品質の方が精度に効く。

## Q3. Agentic RAGのフレームワーク

- **決定**: LangGraph（LangChain系）
- **理由**: マッピング処理は明確なステートマシン。LangGraphのグラフベース制御が設計意図を最も明示的に表現でき、デバッグしやすい。

## Q4. LLM（推論モデル）の選定

- **決定**: 段階的使い分け
- **内訳**:
  - 軽量・高速（検索クエリ生成、関連性判定）: Claude Haiku / GPT-4o-mini
  - 高性能（最終判定、提案文生成）: Claude Sonnet / GPT-4o
- **理由**: 1案件200件×複数ループで数千回のLLM呼び出し。全てを高性能モデルにするとコスト・速度が問題に。

## Q5. PoC（技術検証）の範囲

- **決定**: 実案件PoC（過去RFP 1案件フル処理）
- **内容**: SAP全モジュールのScope Item + モジュール紹介資料をフル投入し、過去の実RFP 1案件分を丸ごと処理
- **正解データ**: 同案件で実際に作成した機能マッピング結果・提案書
- **検証ポイント**: マッピング精度、グラフ探索有効性、提案文品質、処理速度・コスト
