# Round 4: RFPドキュメント取込・分析設計 - Q&A記録

> **ステータス**: 完了
> **日付**: 2026-02-17

## Q1. RFPドキュメントの取込方式

- **決定**: 種別指定アップロード
- **理由**: ユーザー（提案チーム）はRFP構成を理解している。種別指定のコストは低い。

## Q2. ドキュメント解析パイプライン

- **決定**: 形式別パーサー + LLM補完
- **入力形式方針**:
  - 機能要件一覧 → **Excel(.xlsx)**（表構造の正確な保持が必須。PDFに変換すると行列構造が失われる）
  - RFP本体・業務フロー・帳票一覧・補足資料 → **PDF**（パーサー統一でシンプル化）
  - リサーチ情報・DeepResearch結果 → **Markdown(.md)**（LLM親和性が最高）
- **パーサー**: openpyxl(Excel), pdfplumber(PDF), 標準パース(MD) → LLM補完で構造化

## Q3. RFP分析の中間成果物

- **決定**: 中間分析（顧客課題サマリー、評価基準、スケジュール要件、業種・規模情報、構造化済み機能要件一覧）
- **理由**: 後続の提案骨子生成・マッピングの両方に十分な情報量。「行間を読む」推測は含めず、明示的に記載された情報の構造化に留める。

## Q4. ステージゲートの設計

- **決定**: 3ゲート方式
  1. Gate 1: RFP分析結果の確認（顧客課題・重要度の解釈）
  2. Gate 2: マッピング結果の確認（確信度×重要度トリアージ）
  3. Gate 3: 提案構成案の確認（ストーリー・訴求ポイント）

## Q5. DeepResearch（企業情報収集）の統合方式

- **決定**: ハイブリッド
  - 自動: 業種・従業員数・売上高、直近ニュース（企業名入力時にWeb API）
  - 手動: 中計・IR・DeepResearch結果をMarkdownでアップロード
