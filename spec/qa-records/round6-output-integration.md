# Round 6: 出力・外部連携設計 - Q&A記録

> **ステータス**: 完了
> **日付**: 2026-02-17

## Q1. Gamma API連携の設計

- **決定**: 一括生成 + セクション単位再生成
- **理由**: Gamma APIの1リクエスト100Kトークン制限を考慮し、全セクションを一括生成。Gate 3フィードバック後は該当セクションのみ再生成。トライアンドエラーでAPI活用方法を最適化する方針。

## Q2. Excel出力（機能要件回答書）の設計

- **決定**: 両方対応（顧客指定フォーマット書き戻し + 自社フォーマット）
- **理由**: 顧客のRFP Excelフォーマットに回答を埋め込む「書き戻し」と、自社標準のFit&Gap表フォーマットの両方を出力。openpyxlでセル単位書き込み。

## Q3. 見積書生成

- **決定**: 完全手動（スコープ外）
- **理由**: 見積は判断要素が多く自動化のROIが不透明。本システムのスコープはマッピング→提案書生成までとし、見積は従来通り手動作成。マッピング結果（判定レベル・件数）は見積検討の参考情報として利用可能。

## Q4. 案件データ管理・ナレッジ蓄積

- **決定**: DB管理（初期）、将来的にナレッジフィードバックへ拡張
- **内容**:
  - 初期: 案件ごとのRFP・マッピング結果・提案書をDBで構造化管理
  - 将来: 承認済みマッピング結果をナレッジベースにフィードバックし、精度を継続的に向上
- **理由**: まずDB管理で案件データの散逸を防止。ナレッジフィードバックはフィードバックループの設計が重要なため、運用知見が溜まってから段階的に実装。

## Q5. セキュリティ・データ保護

- **決定**: クラウドLLM API（初期）、必要に応じてプライベート環境へ移行
- **内容**:
  - 初期: Claude API / OpenAI APIをクラウド利用（TLS暗号化、APIキー管理、データ保持ポリシー確認）
  - 移行条件: 顧客の機密情報取扱い要件が厳格化した場合にAzure OpenAI等のプライベートエンドポイントへ
- **理由**: 初期はRFP情報（公開入札情報が多い）レベルであり、クラウドAPIで十分。コスト・開発速度を優先。
