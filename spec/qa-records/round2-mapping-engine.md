# Round 2: マッピングエンジンの処理設計 - Q&A記録

> **ステータス**: 完了
> **日付**: 2026-02-17

## Q1. マッピングの検索・照合方式

- **決定**: Agentic RAG（内部でHybrid Search使用）
- **理由**: ERP機能マッピングは1回の検索で正解が出るとは限らない。エージェントが要件を分析→検索→結果不十分なら再検索という「思考しながら検索する」方式が最も精度が高い。NotebookLMとの最大の差別化ポイント。

## Q2. マッピング判定の5段階定義

- **決定**: 実装手段ベース（標準対応 / 設定変更 / アドオン開発 / 外部連携 / 対象外）
- **理由**: 提案段階で最も直感的かつ見積にも直結する分類。明確な判定基準を定義済み。

## Q3. 確信度スコアの算出方式

- **決定**: 検索スコア + LLM自己評価の複合（High/Medium/Low）
- **理由**: LLM単体の自己評価は「自信過剰」になりがち。検索スコアとの複合でバランスを取る。閾値はPoC段階でチューニング。

## Q4. マッピング結果の出力構造

- **決定**: 判定 + 根拠 + 提案文（proposal_text）
- **理由**: マッピングは単なるFit&Gap表作成ではなく、提案書全体の基盤データ。提案文まで生成することで、Fit&Gap表→提案書構成→セクション内容→見積根拠まで一気通貫で活用可能。

## Q5. 人間レビューのワークフロー

- **決定**: 確信度 × 顧客重要度（Must/Should/Could）のトリアージ
- **理由**: 数百件の機能要件のうち、Must×Lowにレビュー工数を集中投下。全件均等レビューは非効率。

## 補足: マッピング→提案の一気通貫思想

マッピング結果は以下の5つの用途に活用される:
1. Fit&Gap表 / 機能要件回答書（直接出力）
2. 提案の強み・弱み分析（訴求ポイント抽出）
3. 提案書アジェンダ・構成最適化（強みの領域を前面に）
4. 提案書セクション内容（proposal_textを素材化）
5. 見積根拠（判定レベル→工数変換）
